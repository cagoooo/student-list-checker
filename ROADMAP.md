## 〇、閱讀認證矩陣與姓名優先校對

### P0｜閱讀認證 / 獎狀名單矩陣格式穩定化

**[難度] 小　[需要] 收集更多實際格式樣本**

已完成第一版：可辨識 `初階`、`中階`、`高階` 這類工作表，並把「班級在欄、姓名在交叉格」的清單攤平成可校對資料。後續建議把這類格式整理成明確的「矩陣名單解析器」策略。

可優化方向：

1. 支援班級欄不只 `101`，也可接受 `一年一班`、`1-1`、`一年甲班`。
2. 支援認證等級不只來自工作表名稱，也可從左上角標題、檔名、合併儲存格推測。
3. 同一檔若混有「矩陣工作表」與「一般清單工作表」，UI 顯示候選來源與筆數。
4. 產生解析摘要：例如「已合併初階 233 筆、中階 110 筆、高階 43 筆」。
5. 若矩陣偵測到少數欄位無班級標題，提示「第 X 欄未辨識班級，已略過」。

價值：閱讀認證、獎狀名單、競賽通過名單常用橫向矩陣，穩定化後可減少人工整理 Excel 的時間。

### P0｜座號降權與「序號 / 座號」語意辨識

**[難度] 小　[需要] 整理常見表格樣態**

已完成第一版：閱讀認證矩陣左側數字不再當座號，而是保留為 `序號`。後續建議把「座號」視為需要確認語意的欄位，而不是看到 1 到 45 的數字就直接認定。

判斷規則建議：

1. 欄名明確是 `座號`、`座次`、`號碼`，才高權重當座號。
2. 欄名是 `編號`、`序號`、`NO`，預設當排序編號，不進入學生座號比對。
3. 矩陣格式左側數字預設當序號，除非同一列或同一區塊明確標示座號。
4. 若同一欄數字連續 1、2、3、4，但姓名分散在不同班級欄，幾乎一定是序號。
5. 若座號欄被系統低信心推測，結果頁顯示「座號僅作輔助」。

價值：避免老師拿排序清單、得獎名單、認證名單來校對時，系統因錯把序號當座號而產生假錯誤。

### P1｜姓名優先校對策略產品化

**[難度] 中　[需要] 依行政情境設計不同模式**

目前已把校對邏輯改成姓名優先：姓名完全相同直接通過；姓名不完全相同時，先看同班姓名近似，再看全校姓名近似；座號只作輔助。下一步可以把它做成明確的「校對模式」。

| 模式 | 適用情境 | 判斷策略 |
|---|---|---|
| 姓名優先 | 閱讀認證、得獎名單、報名清單 | 姓名正確最重要，班級輔助，座號低權重 |
| 班級座號優先 | 座位表、點名表、學籍匯出 | 班級座號與姓名都需一致 |
| 全校姓名搜尋 | 只有姓名的獎狀或活動名冊 | 跨班找同名與近似姓名，要求老師確認 |
| 資料庫稽核 | 更新學生資料庫前 | 重點檢查重複座號、缺漏、轉出空號 |

UI 建議：

1. 上傳後自動判斷模式，並顯示「目前使用：姓名優先校對」。
2. 老師可手動切換模式。
3. 不同模式的錯誤摘要文案不同，避免同一種提示套所有檔案。

價值：同一套平台可以服務多種行政文件，不會因某種文件格式的特性造成大量誤判。

### P1｜問題降噪與可行動摘要

**[難度] 中　[需要] 現有結果資料即可**

這次閱讀認證名單的錯誤多半不是「學生錯了」，而是「座號語意不該被採用」。未來應讓系統先做問題分類與降噪，不要直接把所有低信心比對列成錯誤。

建議分類：

1. **姓名疑似誤植**：姓名近似但不完全相同，需要回原檔確認。
2. **班級可能不符**：姓名存在，但在其他班。
3. **座號忽略**：座號疑似序號，已不列入錯誤。
4. **資料庫缺漏**：上傳名單學生存在於其他來源，但正式資料庫沒有。
5. **格式辨識待確認**：系統不確定某欄是座號、序號或編號。

顯示建議：

```txt
此名單疑似為閱讀認證矩陣，已忽略左側序號。
姓名完全符合：379 筆
姓名疑似誤植：7 筆
需要人工確認：0 筆
```

價值：老師看到的是「下一步要做什麼」，不是一堆技術性錯誤。

### P1｜解析器決策紀錄與除錯面板

**[難度] 中　[需要] 增加 parser metadata**

每次匯入時，系統可以保存一份簡短的解析決策紀錄，方便之後查問題。

建議記錄：

- 選到哪個候選表格。
- 為什麼判定是矩陣格式。
- 哪些欄被判定為班級欄。
- 哪一欄被判定為序號而非座號。
- 各工作表各讀到幾筆。
- 是否有被略過的欄或列。

UI 可以放在「解析詳情」展開區，不干擾一般老師。

價值：未來老師丟來截圖時，可以快速知道是解析器選錯、欄位辨識錯，還是資料庫內容真的不同。

### P2｜同名學生與姓名近似的人工確認流程

**[難度] 中　[需要] UI 互動設計**

姓名優先雖然能降低座號誤判，但會遇到同名學生或近似姓名多候選的情境。建議加入人工確認流程。

功能建議：

1. 若全校有同名學生，顯示候選清單而不是自動挑第一個。
2. 若同班有兩個近似姓名，顯示「可能是 A / B」讓老師選。
3. 老師選定後可套用一次，不一定要寫回正式資料庫。
4. 若老師常常選同一種修正，可記成暫時規則。

價值：保留系統自動化效率，也避免過度自信地自動修錯人。

### P2｜文件格式樣本測試庫

**[難度] 中　[需要] 蒐集匿名化樣本**

目前已有一般名單、ClassSwift 分班檔、閱讀認證矩陣、Word、PDF、OCR 等多種格式。建議建立匿名化測試樣本庫，讓每次改解析器都能回歸測試。

建議樣本類型：

1. 一般直式表格：班級 / 座號 / 姓名。
2. 無標題列名單。
3. 多工作表班級檔。
4. 閱讀認證矩陣。
5. 左側為序號但非座號的清單。
6. 僅姓名清單。
7. Word 表格與合併儲存格。
8. PDF 文字型與掃描型。

價值：後續每次新增功能時，可以避免修好 A 格式卻弄壞 B 格式。

### P2｜老師友善文案與信心說明

**[難度] 小　[需要] 實際回饋**

目前系統已會顯示信心與原因，但可以更白話，尤其是「座號僅作輔助」這類概念。

文案建議：

- 「座號僅作輔助」→「這份表左邊的數字看起來像排序，不像學生座號，所以系統主要檢查姓名。」
- 「姓名近似」→「姓名很接近，可能是其中一個字打錯。」
- 「查無符合姓名」→「正式學生資料庫找不到這個姓名，請確認是否為錯字、舊生、轉入生或資料庫尚未更新。」
- 「班級不符但姓名符合」→「姓名在資料庫中找得到，班級資訊可能是舊資料或此名單不以班級座號為準。」

價值：降低老師看到警示時的緊張感，也更容易知道要回原檔改哪裡。

## 一、學生名單交叉查核與資料版本判讀

### P0｜轉學空號 / 空白預留座號辨識

**[難度] 小　[需要] 確認學校名單慣例**

目前已能略過 ClassSwift 總表中「姓名與 Email 都空白」的預留列，但 `205.xlsx` 這類分班檔會出現 `班級座號` 有值、姓名與信箱空白的列，代表學生已轉學後刻意保留空座號。

建議：
1. 匯入時判斷 `班級座號` 有值但姓名、信箱皆空白，標記為「空號」。
2. 預設不列入錯誤統計，避免老師誤以為名單壞掉。
3. 在結果摘要顯示「略過 2 個空號：20510、20522」。
4. 若使用者需要完整稽核，可提供切換開關「顯示空號列」。

價值：讓新版分班檔與轉學情境更自然，不再需要人工解讀「姓名未填」是不是問題。

---

### P0｜一鍵交叉查核 Firestore 與上傳名單

**[難度] 中　[需要] Firestore students 可公開讀取或後端 callable function**

把這次手動查核流程做成 UI 功能。當結果出現大量錯誤或連續座號錯位時，系統自動產生診斷：

- 該學生是否存在於 Firestore。
- 學號是否存在但座號不同。
- 同名學生是否在其他班或其他座號。
- 該班 Firestore 是否缺某些座號。
- 是否疑似整段座號錯位。

建議顯示範例：

```txt
判斷：資料版本差異
201 班有 19 位學生可用學號對上，但座號與上傳檔不同。
Firestore 缺少：1130004 林俊劼、1130131 游聖恩。
建議：先更新正式學生資料庫，再重新校對。
```

價值：把 Codex 手動交叉查詢變成老師可以直接看的診斷報告。

### P1｜資料來源版本比較器

**[難度] 中　[需要] 支援同時上傳兩份名單或讀取指定資料夾**

支援「A 檔 vs B 檔」比較，例如：

- `二年級各班名單.xls` vs `2年級/205.xlsx`
- ClassSwift 總表 vs 各班最新分檔
- 舊學生資料庫 vs 新學生資料庫

比較維度：

- 學號存在 / 缺漏。
- 姓名變更。
- 班級或座號變更。
- 新增學生、轉出學生、空號。
- 哪份檔案與 Firestore 更一致。

價值：當多個來源版本互相打架時，快速找出「哪份是新資料、哪份是舊資料」。

### P1｜座號錯位偵測與分段摘要

**[難度] 中　[需要] 現有比對結果即可**

當同一班出現多筆低信心姓名錯誤，但學號或姓名可在相鄰座號找到時，系統應判定為「座號錯位」而不是列出一長串單筆錯誤。

規則建議：

1. 同班連續 5 筆以上都能在同班其他座號找到同名 / 同學號。
2. 偏移量固定，例如全部差 1 或差 2。
3. 中間出現缺號時，切成多個錯位區段。

顯示範例：

```txt
201 班疑似座號錯位：
06-23 號多數學生存在於正式資料庫，但座號不同。
可能原因：正式資料庫缺少 03 林俊劼、08 游聖恩，造成後續座號位移。
```

價值：大幅降低錯誤清單噪音，讓老師一眼看到根本原因。

### P1｜正式資料庫更新前預檢報告

**[難度] 中　[需要] admin 上傳學生資料庫流程**

在 admin 把新學生資料推進 Firestore 前，先產生預檢報告：

- 新增幾位學生。
- 移除幾位學生。
- 哪些學生班級 / 座號變更。
- 哪些座號變成空號。
- 是否有同班同座號重複。
- 是否有學號重複或姓名空白。

價值：避免把舊版或錯誤名單匯入正式資料庫，特別適合學期初、轉入轉出後使用。

---

### P2｜空號與轉出學生的資料模型

**[難度] 中　[需要] 決定是否在 Firestore 保存空號**

目前 Firestore 只保存學生，不保存空座號。若行政流程常需要保留座號空缺，可新增 `classSeatStatus` 或在 `studentDatabaseMeta` 保存空號資訊。

可選方案：

1. **不入庫**：只在匯入時略過空號，最簡單。
2. **保存空號清單**：例如 `classSeatGaps/205` 保存 `["10", "22"]`。
3. **保存轉出紀錄**：包含原學生、轉出日期、原因，但需注意個資與權限。

價值：讓系統能區分「漏填姓名」與「已知空號」。

---

### P2｜可信來源優先序設定

**[難度] 小　[需要] 老師確認資料來源規則**

把這次釐清的規則產品化：

- 各班最新分檔（如 `2年級/205.xlsx`）優先於年級總表。
- Firestore 若由最新分檔匯入，應視為正式資料庫。
- ClassSwift 總表可能是較舊版本，只能做參考。

可在 UI 中標示資料來源等級：

```txt
來源可信度：高（各班新版分檔）
來源可信度：中（年級彙整檔）
來源可信度：待確認（未知匯出時間）
```

價值：減少「到底誰是標準答案」的判斷成本。

### P0｜Storage Lifecycle 自動清理 OCR 暫存檔

**[難度] 小　[需要] Firebase project 已建立**

目前 OCR PDF 暫存在 `ocr-jobs/{uid}/{jobId}/source.pdf`，worker 成功或失敗都會主動刪除，但若 worker 異常退出（例如 OOM），暫存檔會殘留。

建議在 GCS 設定 lifecycle rule：
```json
{
  "rule": [{
    "action": { "type": "Delete" },
    "condition": {
      "age": 1,
      "matchesPrefix": ["ocr-jobs/"]
    }
  }]
}
```

這樣即使 worker 異常，暫存檔最多 24 小時後自動清除，符合安全原則。

## 三、OCR 辨識品質提升

### P1｜評估換用 Google Cloud Vision API

**[難度] 中　[需要] GCP project、API 啟用、費用評估**

目前後端 OCR 引擎為 tesseract.js（離線，語言資料打包）。優點是免費、無網路依賴；缺點是對低品質掃描或手寫字準確度有限。

Google Cloud Vision Document AI 的優勢：
- 中文手寫辨識準確度遠高於 tesseract
- 自動偵測表格結構，不需要自己切行列
- 有 1,000 頁/月免費額度（小型學校足夠）

評估建議：先用 tesseract 上線，如果老師回報 OCR 結果太差，再引入 Cloud Vision。切換方式：在 `parseOcrPdf()` 改呼叫 Vision API，行為介面不變，前端完全不用改。

---

### P2｜OCR 後處理：常見辨識錯誤修正

**[難度] 中　[需要] 無**

tesseract 常見辨識錯誤（可建立映射表自動修正）：
- `0` / `O`、`1` / `I`（數字與字母混淆）
- `己` / `已` / `巳`（形近字）
- 座號被辨識成日期格式（如 `01` → `1月`）
- 班級格式破碎（如 `1 年 2 班` 已可正規化，但可再強化）

建議在 `pdfImageOcr()` 產出的文字行後面，套一層「OCR 後修正過濾器」，依常見錯誤 pattern 做 replace，再交給 `splitTextLine()`。

---

### P2｜逐頁進度回報（OCR worker → 前端）

**[難度] 中　[需要] Firebase project 已部署**

目前 `processOcrJob` 的進度是粗粒度的（25% / 75% / 100%）。對多頁 PDF，老師看不出目前到第幾頁。

改善：每辨識完一頁就寫一次 Firestore 更新：
```typescript
await jobRef.update({ progress: 25 + Math.floor((pageNumber / totalPages) * 50) })
```

前端已有即時 snapshot 監聽，不需改前端，只要後端多寫幾次即可。

---

### P2｜OCR Job 超時自動標記 Failed

**[難度] 小　[需要] Firebase project 已部署**

目前 Cloud Functions timeout 設 540 秒，若 worker 超時，Firestore 的 job status 可能停在 `processing`，前端永遠等待。

改善：在 `createOcrJob` 建立 job 時同時記錄 `expiresAt = now + 600s`；可用 Cloud Scheduler 每 10 分鐘掃描 `ocrJobs`，把過期且非 `completed`/`failed` 的 job 標記為 `failed`（附原因：「處理逾時，請重新上傳」）。

## 四、登入與權限系統

### P1｜教師 Google 登入與角色區分

**[難度] 中　[需要] Firebase Auth 已設定、學校 GSuite domain**

建議正式上線後的角色設計：

| 角色 | 條件 | 可做的事 |
|---|---|---|
| 訪客 | 未登入 | 只能用前端示範模式（匿名資料） |
| 教師 | `@mail2.smes.tyc.edu.tw` 登入 | 上傳名單、後端校對、查自己的校對紀錄 |
| 行政 | `admins/{uid}` 存在 | 上述 + 查全校歷史、更新學生資料庫 |
| 資訊組長 | `admins/{uid}` 存在 | 上述 + 管理 admin 名單 |

`assertSchoolCaller()` 已驗證 email domain，只需在 Firestore 建立 `admins` 集合即可啟用角色分流。

### P1｜前端顯示目前使用者的校對歷史

**[難度] 中　[需要] Firebase 已部署**

每次後端校對都已寫入 `validations/{id}`，包含上傳者 uid、檔名、摘要、問題清單預覽。

可在前端加入「我的校對紀錄」頁面：
- 列出最近 10 次
- 點開可看問題清單預覽
- 顯示上傳時間、檔名、通過 / 錯誤筆數

## 五、校對準確度提升

### P1｜擴充中文姓名校正字典

**[難度] 小　[需要] 實際誤植案例回饋**

目前 `variantGroups`（異體字）和 `similarGroups`（形近同音字）已有基礎字組，但石門國小實際名冊可能有特殊罕見字。

建議建立回饋機制：
1. 老師發現系統「沒有幫他找出應該很像的名字」時，可提交回饋
2. 由資訊組長定期匯整，更新 `variantGroups` / `similarGroups`
3. 可加入注音 / 拼音相似度（兼顧同音字更廣的覆蓋）

---

### P2｜後端支援 `.xls` 舊格式

**[難度] 小　[需要] 無**

目前後端 `parseRosterFile()` 支援 `.xlsx`（ExcelJS），但不支援舊版 `.xls`。前端有走 `xlsx` 套件的 fallback，但前端解析安全性有疑慮。

改善：在 functions 加入 `xlsjs`，或把 `.xls` 轉 buffer 後用 ExcelJS 處理，讓 `.xls` 也完整走後端管線。

---

### P2｜姓名相似度說明更友善

**[難度] 小　[需要] 無**

目前「提示」欄顯示術語（異體字、形近字），對老師較難理解。

建議改成白話說明：
- 「異體字」→「這兩個字意思相同，但寫法不同，可能是字型問題」
- 「形近字」→「這兩個字長得很像，可能是 KEY 錯」
- 「缺一字」→「姓名少了一個字，請回原檔確認」

---

### P3｜全校名冊跨班搜尋模式

**[難度] 中　[需要] 無**

進階需求（行政常見情境）：
- 上傳只有姓名（無班級座號）的得獎名單
- 系統跨班找出所有相符或相近的學生讓老師確認

這個模式比現有流程更寬鬆，對行政處名冊整合很有幫助。

---

## 六、使用者體驗改善

### P1｜問題表格篩選 + 分頁

**[難度] 小　[需要] 無**

若一份名單有數十筆問題，目前表格是一長頁。

建議：
- 問題數 > 20 時顯示「顯示前 20 筆 / 共 N 筆」，附「載入全部」按鈕
- 加入問題類型篩選（只看「錯誤」/ 只看「待確認」）
- 表格加上 sticky 表頭，方便對照欄位

---

### P1｜問題清單一鍵複製成純文字報告

**[難度] 小　[需要] 無**

老師常需要把問題清單貼到 Line 群組或 email 通知。

建議加一個「複製問題清單」按鈕，輸出格式如：
```
📋 名單校對結果（2026-06-26）
檔名：106年校慶得獎名冊.xlsx
共 45 筆，通過 42 筆，需確認 2 筆，錯誤 1 筆

【待確認】第 12 列：3年1班 05號 王曉明
  → 系統建議：王小明（高信心，字形相似）

【錯誤】第 28 列：5年3班 99號 李大華
  → 座號不存在，請確認
```

---

### P2｜拖曳上傳 / 複製貼上支援

**[難度] 小　[需要] 無**

目前只支援點按「上傳名單」按鈕。

建議加入：
1. 拖曳檔案到上傳區（drag & drop）
2. 從 Excel 複製一段資料後，直接貼上到系統自動解析（clipboard paste）

這對習慣用滑鼠操作 Excel 的老師特別方便。

### P2｜行動版體驗優化

建議：

使用 `prefers-color-scheme: dark` media query + CSS 變數切換，對夜間在辦公室操作的行政人員友善。

### P3｜無障礙改善（WCAG 2.1 AA）

## 七、安全性改善

### P1｜替換前端 xlsx 套件

**[難度] 中　[需要] 無**

`xlsx`（SheetJS）套件有 high severity security audit 警告，官方社群版長期未修補。

替換選項：
- **ExcelJS**：後端已使用，前端可引入（但體積較大）
- **SheetJS Pro**：官方付費版有修補，但費用需評估
- **✅ 建議：改走後端解析**（登入後全部走 `validateRosterFileOnBackend()`，移除前端 `xlsx` 依賴；未登入的示範模式不上傳真實資料，風險可接受）

---

### P1｜Firestore 安全規則補強

**[難度] 小　[需要] Firebase project 已部署**

目前預設拒絕，上線前建議補充：
- `validations`：只有 `createdByUid == request.auth.uid` 可讀自己的紀錄；admin 可讀全部
- `ocrJobs`：同上
- Functions 層加上 uid 寫入頻率保護（防止惡意大量送工作）

---

### P2｜後端輸入驗證強化

**[難度] 小　[需要] 無**

目前已限制 row 數量上限（2000）。可再加：
- 每個 cell 長度上限（防止超長字串）
- 姓名欄位只允許 CJK + 空白 + 少數符號
- 檔名只允許合法副檔名（已部分實作，可再補全）

---

## 八、管理功能

### P2｜學生資料庫版本管理

**[難度] 中　[需要] Firebase project 已部署**

目前每次更新學生資料庫會直接覆蓋 Firestore。建議改為有版本記錄：
- `studentDatabaseVersions/{versionId}`：儲存上傳者、時間、來源檔名、學生數
- 可回退到上一版（匯錯資料時）
- 前端顯示目前版本、最後更新時間

**[難度] 中　[需要] Firebase project 已部署**

- 可依日期、班級、上傳者篩選
- 匯出整份 CSV（用於期末審計）

每週自動統計「本週各班上傳名單的常見錯誤類型」，以 email 發給行政主任，讓學校主動追蹤哪些老師常 KEY 錯名字。

## 九、進階功能（長期）

### P3｜支援圖片格式上傳（JPG / PNG）

**[難度] 中　[需要] Cloud Vision 或 tesseract.js**

目前 OCR 僅支援 PDF。很多老師可能直接用手機拍照上傳。

改善：允許 `.jpg`、`.png` 上傳，後端用 tesseract.js 或 Cloud Vision 直接辨識，不需要 PDF 包裝層。

---

### P3｜獎狀 / 報名表模板輸出

**[難度] 大　[需要] 無**

校對完成後，讓老師選擇模板，自動套入正確學生資料輸出獎狀或報名表 PDF。

注意：這是全新功能，不在目前主線。建議先把校對準確度做穩再開發。

---

### P3｜多校共用（推廣到龍潭區其他學校）

**[難度] 大　[需要] 架構重構**

若要推廣到其他學校：
- 學生資料庫多租戶隔離（`schools/{schoolId}/students`）
- 登入後根據 email domain 決定讀哪個資料庫
- 管理介面支援跨校
