# 未來優化改良與可開發功能建議清單

更新日期：2026-06-26

> 產品方向校正（2026-06-26）：本工具正式主線不是「幫老師產生修正版檔案」，而是「老師上傳檔案後，後端自動辨識並用正式學生資料庫比對，前端直接回報整份名單是否正確、哪些學生姓名可能 KEY 錯或資料不符」。下載修正版 Excel / Word 只保留為輔助能力，不再作為核心工作流。

> 進度註記（2026-06-26）：Excel / CSV / PDF / Word 已共用統一匯入管線（`src/lib/importer`），含欄位偵測信心分數與匯入診斷；建置流程已加入版本標記與 git 強化（`scripts/finalize-build.mjs`）。匯入能力再深化：PDF 頁碼 / 多欄、掃描 PDF OCR（tesseract.js）、Word 合併儲存格 / 段落、`.doc` best-effort 還原與「輸出修正版 Word」皆已上線，但輸出修正版已降為輔助能力。以下清單已依現況調整，標示「已落地」者代表第一版完成、仍可深化。

## 一、資料庫與後台管理

### 1. 正式學生資料庫

建議將目前的 `src/data/students.json` 移到正式後端資料庫，例如 Firebase Firestore、Supabase PostgreSQL、Cloud SQL 或校內既有資料庫。

可開發功能：

- 學生基本資料表：學號、姓名、年級、班級、座號、性別、狀態
- 學年度與學期欄位：避免新舊學年度資料混用
- 轉入、轉出、休學、畢業狀態
- 班級代碼標準化，例如 `601`、`6年1班`、`六年甲班` 對應同一班
- 資料更新版本紀錄

建議優先度：P0

### 2. 後台資料更新

目前前端已可上傳學務系統 `.xls` 更新本機資料庫，下一步可改成後台管理功能。

可開發功能：

- 管理員上傳學務系統匯出的學生資料概況
- 上傳前預覽新增、修改、刪除筆數
- 更新前二次確認
- 自動保留前一版資料，可一鍵回復
- 顯示最後更新時間與更新人
- 匯入失敗時提供錯誤列號與原因

建議優先度：P0

### 3. API 設計

建議把檔案辨識、欄位判定、學生資料庫比對與錯誤分類移到後端，前端只接收校對結果摘要與問題清單。這樣老師不必下載修正版，也不必在前端處理複雜文件流程。

建議 API：

```txt
POST /api/student-database/import
GET  /api/students
POST /api/uploads
POST /api/validations
GET  /api/validations/:id
```

核心回傳資料應包含：

- `summary`：總筆數、通過、待確認、錯誤數量，以及整份名單是否可放心使用
- `issues`：只列出需要老師注意的學生，例如姓名疑似 KEY 錯、班級 / 座號不符、查無此人
- `source`：原始檔名、頁碼 / 列號 / 段落位置
- `suggestion`：系統比對到的正確學生資料與信心分級

建議優先度：P0

## 二、名單解析與辨識能力

### 4. Excel / CSV 解析安全化

目前使用 `xlsx` 套件，`npm audit` 顯示 high severity 且無官方修補版。正式上線前建議處理。

可行方向：

- 將解析移到後端隔離環境
- 限制檔案大小與副檔名
- 掃描工作表數量與資料列數
- 改用維護狀態較好的解析方案
- 上傳檔案只保留短時間，處理完即刪除

建議優先度：P0

### 5. PDF 解析

PDF 常見於老師或處室提供的活動名冊，但版面差異大，建議分階段做。

第一階段（已落地）：

- 抽取純文字 ✅
- 偵測班級、座號、姓名模式 ✅
- 顯示抽取結果讓使用者確認欄位 ✅（匯入診斷）

第二階段（已落地）：

- 偵測多欄版面 ✅（依文字項寬度判斷欄位間距）
- 保留原始頁碼與位置，方便標錯 ✅（校對結果列顯示「來源」頁碼）
- 待補：明確表格格線版面的精準切欄

第三階段（已落地，可再提升）：

- 掃描 PDF 或影像 PDF 加 OCR ✅（tesseract.js chi_tra+eng，動態載入）
- 待補：低品質掃描、直式文字、欄位錯位的準確度，與中文字典後處理

建議優先度：剩餘精修 P2 / P3

### 6. Word 解析

Word 檔常有表格、條列、複製貼上的名單格式。

已落地：

- 讀取 `.docx` 表格 ✅
- 讀取段落中的名單 ✅
- 偵測「班級 座號 姓名」混寫格式 ✅

已落地（延伸）：

- 合併儲存格（colspan / rowspan）網格還原 ✅
- 段落 / 條列型名單抽取 ✅
- `.doc`（舊格式）best-effort 文字還原 ✅
- 輸出修正版 Word（`.docx`）✅

可深化：

- 保留原表格樣式 / 框線輸出修正版 Word（目前為純表格）
- `.doc` 還原準確度提升（OLE 結構化解析）

建議優先度：P3

### 7. 欄位智慧對應

目前已有基本欄位偵測，未來可加強不同老師的欄名習慣。

可辨識欄名：

- 班級、班別、班、年班、class
- 座號、號碼、編號、座位、seat、number
- 姓名、學生姓名、名字、學生、name
- 活動、項目、組別、獎項、備註

已落地：

- 自動記住使用者常用欄位對應 ✅（`columnMemory.ts`，依表頭簽章）
- 多工作表選擇 ✅（候選表格選擇器）
- 合併儲存格處理 ✅（Word colspan/rowspan 網格還原）
- 第一列不是欄位名稱時，自動尋找欄位列 ✅，且無標題列也能整份視為資料
- 辨識失敗逃生口 ✅（手動重指定標題列 / 標記無標題列）

可深化：

- 依「整欄資料樣態」再強化欄位對應（目前已用內容推測，可加更多欄型）
- 跨檔案的欄位對應雲端同步（目前存 localStorage）

建議優先度：剩餘深化 P1

## 三、校對規則與修正建議

### 8. 錯誤分類更細緻

目前有通過、待確認、錯誤三類，未來可提供更精準的錯誤原因。

建議分類：

- 班級正確、座號正確、姓名錯字
- 姓名正確、班級錯誤
- 姓名正確、座號錯誤
- 班級不存在
- 座號不存在
- 同名學生，需要人工確認
- 缺少必要欄位
- 名單中重複學生
- 同一學生出現在多個項目

建議優先度：P1

### 9. 模糊比對強化

目前使用簡單字串距離，未來可加上更符合中文姓名的比對規則。

已落地：

- 中文姓名校正引擎第一版 ✅（`src/lib/importer/nameMatch.ts`）
- 常見異體字或近形字提示 ✅（如 陳/陈、峯/峰、堃/坤、玟/雯、宥/侑）
- 少一字、多一字、顛倒字提示 ✅
- 姓名中空白與常見分隔符正規化 ✅
- 信心分數分級：高、中、低 ✅

可開發功能：

- 依實際校務資料持續擴充同音字、形近字與新舊字形對照
- 新舊字形對照
- 將低信心案例做成可回饋字典的人工確認流程

建議優先度：第一版已落地；字典與人工確認流程 P1

### 10. 人工確認流程

學生姓名校正不建議完全自動覆蓋，應保留人工確認。

可開發功能：

- 每列「接受建議」、「忽略」、「手動修改」
- 批次套用高信心建議
- 低信心建議必須人工確認
- 已確認列鎖定，避免再次變動
- 下載前檢查是否仍有未處理錯誤

建議優先度：P1

### 25. 建置、版本與 CI 強健性

延續這次 `scripts/finalize-build.mjs` 的改良（git 取版號加上 `safe.directory` 與明確 stdio、避免 CI 或不同擁有者目錄報 dubious ownership），可把「版本可追溯、建置可重現」做成固定基礎建設。

已落地：

- 建置自動寫入 app 版本、`sw.js` 版本與 `version.json` ✅
- finalize-build 的 git 呼叫不再因目錄擁有權或互動式輸入而失敗 ✅

可開發功能：

- 把 finalize-build 的版號取得抽成單一可測函式，加上單元測試（git 不可用 / 無 SHA / CI 帶 `GITHUB_SHA` 三種情境）
- CI 加上「建置產物驗證」步驟：檢查 `dist/index.html`、`dist/sw.js`、`dist/version.json` 是否已成功替換掉 `__APP_VERSION__`、`__BUILD_VERSION__` 等佔位字串（沒換成功就讓 CI 失敗）
- 部署後自動拉取線上 `version.json` 比對，確認上線版本與本次 commit 一致（避免「deploy 完還是舊版」）
- 在頁面或主控台顯示目前版本與 build 時間，方便老師回報問題時對版本
- `npm audit` / 相依套件弱點掃描納入 CI，特別盯 `xlsx` 等解析套件（呼應第 4 點）
- 鎖定 Node 版本（`.nvmrc` 或 `engines`），避免 Windows + Node 版本差異造成的建置不一致
- 建置產物大小監看，PDF / Word 解析套件讓 bundle 變大時提早發現

建議優先度：P1

## 建議開發路線

### 階段 1：內部可用版

目標：讓老師上傳檔案後，系統能穩定回報名單是否正確。

- 建立 Firebase project 並設定 GitHub Secrets
- 部署 Firestore Rules
- 建立第一位 `admins/{uid}` 管理員
- 將學生資料同步到 Firestore
- 後端化 Excel 解析
- 後端回傳校對摘要與問題清單
- 前端改成「全部正確 / 有問題」的結果回報頁

### 階段 2：行政流程版

目標：讓名單校對成為處室固定工作流，不讓老師多做下載與改檔動作。

- 任務狀態
- 老師上傳、行政退回、重新上傳
- 錯誤報告
- 校對紀錄與問題回報

### 階段 3：文件產出版

目標：在確認名單正確後，再視需求延伸行政文件產出。

- 獎狀合併列印資料
- 報名表 / 名冊模板
- 錯誤報告 / 留存紀錄輸出
- 下載紀錄與封存

### 階段 4：智慧辨識版

目標：支援更多不規則來源格式。

- PDF 文字抽取
- Word 表格解析
- OCR
- 中文姓名進階模糊比對
- 原始文件位置標註
