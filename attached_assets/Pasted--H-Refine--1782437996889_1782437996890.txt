產品方向：
這不是「修正版文件產生器」。
核心目標是：
老師上傳檔案後，系統在後端辨識並用正式學生資料庫比對，前端只回報：
1. 這份名單是否全部正確
2. 哪些學生姓名可能 KEY 錯
3. 哪些班級 / 座號 / 姓名對不起來
4. 哪些資料查無此人

老師不需要下載修正版，也不需要在平台上編輯名單。

重要安全原則：
- 不要把原始 PDF 存進 Firestore。
- 原始 PDF 只能短期存在 Storage path。
- worker 成功或失敗都應盡量刪除暫存檔。
- 校對紀錄只保存摘要與問題清單，不保存完整原始名單。
- 掃描 OCR 比較重，應用背景 job，不要讓老師畫面同步等待。

目前 OCR worker 狀態：
- worker 管線已完成。
- 目前 `parseOcrPdf()` 仍主要使用 PDF 文字抽取。
- 若 PDF 是純影像掃描、沒有文字層，會失敗並提示：
  「OCR worker 管線已建立，但目前尚未接入影像 OCR 引擎；這份 PDF 沒有可抽取的文字。」
- 下一步重點是接入真正影像 OCR 引擎。

下一步建議：
1. 實作真正影像 OCR 引擎
   可評估兩條路：
   - 短期：Functions 端接 `tesseract.js` + PDF render，但要注意 Cloud Functions Node 22、中文字庫、冷啟動、記憶體與部署體積。
   - 穩定正式版：接 Google Cloud Vision 或 Document AI，但需要正式 GCP project、API 啟用、費用控管與 IAM。
2. 優先保持漸進式：
   - 不要把大 OCR 同步塞進 callable function。
   - 影像 OCR 應接在 `processOcrJob` / `parseOcrPdf()` 內。
   - OCR 產出 rows 後繼續走既有 `rowsToRosterRows()` 與 `validateRows()`。
3. 前端可再補：
   - OCR job 完成後，如果 result issues 很多，要確認 UI scroll / table 表現。
   - 可加「重新上傳較清晰檔案」提示。
   - 可設計 job timeout / 長時間處理提示。
4. 後端可再補：
   - Storage lifecycle / TTL 清理策略。
   - failed job 的暫存檔清理補償機制。
   - OCR job Firestore index 如未來要列表查詢再補。
   - resultIssues 若大量，要考慮只保存 issue，不保存完整 rows；目前符合此原則。
